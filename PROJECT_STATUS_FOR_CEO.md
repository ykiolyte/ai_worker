# Отчет о состоянии проекта ai_worker

Дата подготовки: 15 мая 2026  
Статус: локальный MVP поднят, основные сервисы работают, проект готов к функциональному тестированию и демонстрации текущих возможностей.

## 1. Резюме для руководителя

`ai_worker` - это MVP системы автоматизации поиска товаров и поставщиков. Система позволяет запускать поисковые запросы, собирать карточки товаров, извлекать контакты поставщиков, сравнивать поставщиков по метрикам, готовить коммуникацию с поставщиком и использовать локального AI-ассистента на карточке товара.

На текущем этапе проект уже не является только прототипом интерфейса: есть backend API, web-интерфейс, worker-процессы, локальная модель через Ollama, браузерный исследовательский слой, поисковый слой, тестовый supplier-site, почтовая инфраструктура для тестирования и набор автотестов. Проект развернут через Docker Compose и сейчас находится в рабочем состоянии.

Ключевая ценность текущей версии: пользователь может пройти основной сценарий от поиска товара до анализа поставщика и подготовки дальнейших действий. При этом часть функций пока остается MVP-уровня и требует повышения надежности перед промышленным использованием.

## 2. Текущее состояние системы

### Запущенные сервисы

В Docker Compose подняты следующие сервисы:

- `backend` - API приложения, доступен на `http://localhost:8000`;
- `webui` - пользовательский интерфейс, доступен на `http://localhost:5173`;
- `worker` - фоновая обработка задач поиска и коммуникации;
- `postgres` - база данных;
- `redis` - очередь/служебное хранилище;
- `ollama` - локальный AI runtime;
- `browser-mcp` - браузерный слой для исследования сайтов;
- `searxng` - дополнительный поисковый движок;
- `supplier-site` - тестовый сайт поставщика для E2E-проверок;
- `mailpit` - тестовая почтовая инфраструктура.

Проверка `http://localhost:8000/health` возвращает статус `ok`.

### Архитектура

Проект построен как локально разворачиваемый продуктовый MVP:

- `frontend` - React/Vite интерфейс для управления поисками, каталогом товаров, карточкой товара и коммуникацией;
- `backend` - FastAPI API, бизнес-логика, сериализация карточек, управление задачами и контактами;
- `worker` - фоновые задачи поиска товаров, обработки контактов и коммуникации;
- `connectors` - интеграции с браузером, поиском, email, Telegram и локальной AI-моделью;
- `tests` - контрактные и интеграционные тесты;
- `openspec` - спецификации изменений и задач;
- `docker-compose.yml` - локальная инфраструктура для полного запуска.

## 3. Что уже сделано

### 3.1. Базовый MVP product sourcing

Реализован основной контур:

- создание поискового запроса;
- указание лимита результатов;
- создание карточек товаров;
- отображение каталога найденных товаров;
- просмотр карточки товара;
- хранение контактов поставщика;
- запуск коммуникации с поставщиком;
- отображение истории сообщений.

### 3.2. Поиск реальных поставщиков

Добавлены улучшения для поиска не только демонстрационных карточек:

- multi-engine поиск;
- поддержка DuckDuckGo/SearxNG;
- расширение стратегий поиска;
- обработка частичных сбоев поисковых движков;
- дедупликация результатов;
- повышение лимита результатов;
- извлечение контактов из HTML, включая типовые разделы вроде `contact`, `contacts`, `about`, `support`, footer-блоки.

### 3.3. Сравнение поставщиков

Добавлена оценка поставщиков по нескольким практическим метрикам:

- цена;
- качество контакта;
- доступность контакта;
- полнота данных;
- наличие ответа от поставщика;
- качество коммуникации;
- прослеживаемость источника.

На базе метрик рассчитывается общий рейтинг поставщика. Это помогает быстрее отбирать перспективные варианты и не сравнивать карточки вручную.

### 3.4. AI-помощь в коммуникации

Реализованы функции AI для работы с поставщиком:

- генерация первого сообщения поставщику;
- генерация ответа в существующем диалоге;
- учет языка и стиля сообщения;
- анализ входящего ответа поставщика;
- извлечение условий: цена, MOQ, сроки, доступность, оплата, доставка;
- автоматическое сохранение анализа в карточку товара.

### 3.5. Внутренний AI Assistant на карточке товара

Добавлен внутренний AI Assistant, который не отправляет сообщения поставщику и работает как помощник пользователя:

- ассистент доступен на странице карточки товара;
- чат открывается боковой панелью, карточка товара остается видимой;
- история переписки с ассистентом сохраняется в карточке и не пропадает после обновления страницы;
- быстрые подсказки помогают спросить о рисках, следующем шаге, слабых местах и черновике письма;
- JSON-подобные ответы модели преобразуются в читаемый текст для пользователя.

### 3.6. Экспорт данных

Добавлена возможность сохранить информацию по товару/поставщику в Excel-совместимом формате. В экспорт включаются основные данные карточки, контакты, рейтинг, метрики и извлеченные AI-условия.

### 3.7. Локальный AI runtime

Проект переведен на работу с локальной моделью через Ollama. Для текущего железа пользователя выбран практичный подход с моделью семейства Qwen 2.5 в диапазоне, который должен быть реалистичен для RTX 3050, 32 GB RAM и i5-12500.

Цель такого подхода - сохранить автономность, снизить зависимость от внешних API и дать возможность тестировать AI-сценарии локально.

### 3.8. DevOps и запуск

Подготовлен локальный запуск через Docker Compose:

- backend;
- frontend;
- worker;
- Postgres;
- Redis;
- Ollama;

Есть `.env` для локальной конфигурации. Секреты не должны попадать в отчеты, git или публичные файлы.

## 6. Текущие трудности и риски

### 6.1. Качество поиска поставщиков

Поиск реальных поставщиков зависит от внешних сайтов и поисковых движков. Возможные проблемы:

- сайты блокируют автоматизированный доступ;
- часть страниц закрыта Cloudflare, captcha или anti-bot защитой;
- поисковики могут возвращать маркетплейсы вместо прямых поставщиков;
- не все сайты явно публикуют email/Telegram;
- контакты часто спрятаны в футере, на отдельной странице или под JavaScript.

Риск для бизнеса: пользователь может ожидать 20 качественных поставщиков, но фактически получить меньше, если внешние источники ограничены.

### 6.2. Captcha и anti-bot ограничения

В проекте допускается обход только простых, легитимных препятствий уровня cookie banner, простых interstitial-страниц и базовых повторных попыток. Агрессивный обход защит, взлом, обход платных стен и нарушение правил сайтов не должны быть частью продукта.

Рекомендуемый подход: сделать систему устойчивой к отказам и расширять пул легальных источников данных, а не пытаться силой проходить все блокировки.

### 6.3. Локальные AI-модели

Локальная модель дает автономность, но имеет ограничения:

- качество ответа ниже, чем у сильных облачных моделей;
- возможны JSON-подобные или технические ответы;
- скорость зависит от железа;
- RTX 3050 ограничивает размер модели и скорость инференса;
- требуется аккуратная настройка prompt и постобработка.

Часть проблемы уже закрыта: ответы внутреннего AI Assistant приводятся к читаемому виду.

### 6.4. Работа с email

Email-интеграции требуют аккуратной настройки:

- секреты должны храниться только в `.env`;
- реальные SMTP/IMAP доступы нельзя коммитить;
- автоответы должны иметь режим согласования, чтобы избежать несанкционированной отправки;
- нужен аудит отправленных сообщений.

### 6.5. Состояние репозитория

В рабочем дереве есть незакоммиченные изменения. Это нормально для активной разработки, но перед передачей/релизом нужно:

- сгруппировать изменения по смысловым коммитам;
- убедиться, что `.env` не попадает в git;
- проверить `.env.example` на отсутствие секретов;
- архивировать завершенные OpenSpec changes;
- зафиксировать версию, которую демонстрируем руководству.

## 9. Рекомендуемый следующий фокус разработки

Самое безопасное направление для продолжения - улучшение качества поиска и извлечения контактов поставщиков.

Причина: бизнес-ценность системы напрямую зависит от того, сколько реальных поставщиков пользователь получает и насколько быстро может с ними связаться. Интерфейс, AI Assistant, рейтинг и экспорт уже дают рабочий контур, но качество входных данных остается главным ограничителем.

Приоритеты:

1. Извлечение контактов из HTML и связанных страниц.
2. Ранжирование прямых поставщиков выше маркетплейсов.
3. Увеличение количества релевантных результатов.
4. Повышение прозрачности: почему поставщик найден, откуда контакт, насколько ему можно доверять.

## 10. Вывод

Проект находится в состоянии рабочего локального MVP. Он уже демонстрирует ключевой бизнес-сценарий: поиск товара, сбор поставщиков, оценка, коммуникация, AI-поддержка пользователя и экспорт результата.

Основной риск сейчас не в запуске инфраструктуры, а в качестве внешних данных: реальные сайты поставщиков неоднородны, часто защищены и не всегда публикуют контакты явно. Поэтому следующий этап должен быть направлен на повышение надежности supplier discovery и качества contact extraction.

С технической точки зрения проект готов к дальнейшему тестированию и постепенной стабилизации. Перед показом генеральному директору желательно зафиксировать демонстрационный сценарий и не использовать случайные запросы как единственный способ презентации, потому что внешние поисковые источники могут давать нестабильный результат.
