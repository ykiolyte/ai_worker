## ADDED Requirements

### Requirement: Request Supplier Contact

The system SHALL allow a user to request that the agent contacts a supplier for a product.

#### Scenario: User requests supplier contact

- GIVEN a product has at least one supported supplier contact
- WHEN the user clicks "Contact supplier"
- THEN the system SHALL create a contact attempt
- AND the contact attempt status SHALL be `queued`
- AND the system SHALL create an agent task of type `supplier_contact`
- AND the API response SHALL return without waiting for message delivery

#### Scenario: Product has no supplier contacts

- GIVEN a product has no supplier contacts
- WHEN the user opens the product details page
- THEN the system SHALL disable the "Contact supplier" action
- AND the system SHALL explain that no supplier contact is available

#### Scenario: Active contact attempt already exists

- GIVEN a product has a contact attempt with status `queued` or `running`
- WHEN the user tries to create another supplier contact attempt for that product
- THEN the system SHALL reject the duplicate active attempt
- AND the UI SHALL keep the contact action disabled while the attempt is active

### Requirement: Select Connector By Contact Type

The system SHALL select the communication connector based on supplier contact type.

#### Scenario: Contact type is email

- GIVEN a supplier contact has type `email`
- WHEN the agent processes the supplier contact task
- THEN the system SHALL use the email connector

#### Scenario: Contact type is Telegram

- GIVEN a supplier contact has type `telegram`
- WHEN the agent processes the supplier contact task
- THEN the system SHALL use the Telegram connector

### Requirement: Persist Contact Attempt Result

The system SHALL persist the result of each supplier contact attempt.

#### Scenario: Message sent successfully

- GIVEN the agent successfully sends a supplier message
- WHEN the connector returns success
- THEN the system SHALL update the contact attempt status to `sent`
- AND the system SHALL persist the sent message text
- AND the system SHALL persist external message identifier when available

#### Scenario: Message sending fails

- GIVEN the agent attempts to contact a supplier
- WHEN the connector returns an error
- THEN the system SHALL update the contact attempt status to `failed`
- AND the system SHALL persist a user-readable error message

### Requirement: Model-Generated Safe Supplier Message

The system SHALL generate initial supplier messages with the configured `ModelProvider` and SHALL NOT replace model output with hardcoded outbound message templates. Supplier messages SHALL request information without creating purchase, payment, or legal commitments.

#### Scenario: Supplier message is generated

- GIVEN the agent prepares a supplier contact message
- WHEN the message is validated before sending
- THEN the message SHALL be generated by the configured model provider
- THEN the message SHALL include the product name and product URL
- AND the message SHALL ask about price, availability, minimum order quantity, delivery timing, payment, and delivery terms
- AND the message SHALL NOT confirm an order, promise payment, request policy bypass, or transmit payment data

#### Scenario: Initial model output cannot be used

- GIVEN the model returns an empty, incomplete, or unsafe initial supplier message
- WHEN the system validates the generated message before sending
- THEN the system SHALL request a corrected supplier message from the model provider
- AND the system SHALL NOT send a hardcoded template or generic fallback message

### Requirement: Contextual Supplier Conversation Replies

The system SHALL generate follow-up supplier replies with the configured `ModelProvider` using the complete available conversation context.

#### Scenario: Supplier asks a follow-up question

- GIVEN a supplier has replied to an earlier outbound message
- WHEN the system generates an agent reply
- THEN the reply SHALL be generated by the configured model provider
- AND the prompt SHALL include the latest supplier message, prior conversation messages, the product title, and the product URL
- AND the reply SHALL answer the supplier's latest question directly
- AND the system SHALL NOT replace the model output with a generic fallback reply

#### Scenario: First model output cannot be used

- GIVEN the model returns an empty, too short, repeated initial outreach, or unsafe reply
- WHEN the system validates the generated reply before sending
- THEN the system SHALL request a corrected contextual reply from the model provider
- AND the system SHALL NOT send a generic fallback reply

### Requirement: Automatic Contextual AI Replies

The system SHALL automatically answer matched inbound supplier Gmail replies using product and conversation context while preserving MVP safety limits.

#### Scenario: Supplier replies over Gmail

- GIVEN inbound Gmail sync matches a supplier message to a product conversation
- WHEN the message is stored
- THEN the system SHALL generate an AI reply using conversation context
- AND the system SHALL send the generated reply through the selected connector
- AND the system SHALL persist the outbound reply as part of the conversation

#### Scenario: Supplier asks for an order, payment, or legal commitment

- GIVEN inbound Gmail sync matches a supplier message that asks for order confirmation, payment, payment details, or legal requisites
- WHEN the system prepares the reply
- THEN the system SHALL answer without confirming an order, promising payment, sending payment details, or creating legal commitments
- AND the system SHALL request a corrected contextual reply from the model if the first model output cannot be used
